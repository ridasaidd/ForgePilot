# FP-006 — Task Classification Standards

## Task

Define standards for classifying ForgePilot packets by task type, risk, constraint strictness, evidence sensitivity, blast radius, and routing relevance.

## Goal

Ensure every packet can be classified before execution so future routing decisions are based on comparable evidence.

FP-006 answers one question:

What kind of work is this packet asking a model to perform?

This packet defines standards only.

It does not add SQLite storage, CLI behavior, routing logic, model selection, automation, aggregation, dashboards, reporting, or schema implementation.

## Governing Principles

This packet is constrained by:

- P01 — ForgePilot records observations, not narratives.
- P02 — Trust cannot be retroactively created.
- P03 — ForgePilot does not optimize for favorable outcomes.
- P04 — Only admitted evidence may influence observatory outputs.
- P05 — Do not build infrastructure for evidence that does not yet exist.
- P06 — Classification follows observation.

## Core Model

Every packet must be classifiable across independent axes.

These axes must not be collapsed into one field.

A packet may be low-risk but high-complexity.

A packet may be documentation-only but trust-critical.

A packet may be implementation-heavy but low evidence sensitivity.

A packet may be exploratory and therefore unsuitable for automatic routing.

## Required Classification Axes

FP-006 must define standards for at least the following axes:

1. Task Class
2. Risk Level
3. Constraint Strictness
4. Evidence Sensitivity
5. Expected Blast Radius
6. Primary Skill Required
7. Audit Requirement
8. Challenger Requirement
9. Routing Eligibility

## Task Class

Task Class describes the kind of work requested by the packet.

The standard must define a controlled vocabulary.

At minimum, include:

- STANDARDS
- DOCUMENTATION
- PERSISTENCE
- SCHEMA
- VALIDATION
- TELEMETRY
- CLI
- TESTING
- REFACTOR
- BUG_FIX
- AUDIT
- ROUTING
- RESEARCH
- WORKFLOW
- UNKNOWN

Task Class must describe the packet’s primary work, not the files touched.

If a packet spans multiple task classes, it may define a primary task class and secondary task classes.

Task Class must not be inferred from model output after execution.

It must be assigned before execution and may be corrected only by an explicit classification observation.

## Risk Level

Risk Level describes the potential damage if the packet is implemented incorrectly.

The standard must define at least:

- LOW
- MEDIUM
- HIGH
- CRITICAL

Risk Level must consider:

- data integrity
- trust boundary impact
- audit impact
- persistence impact
- routing impact
- security or safety impact
- likelihood of corrupting future evidence

A packet that changes evidence admission, trust tiers, telemetry ingestion, or routing behavior must not be LOW risk.

## Constraint Strictness

Constraint Strictness describes how tightly the executor must stay within the packet.

The standard must define at least:

- LOOSE
- NORMAL
- STRICT
- FROZEN

STRICT means extra behavior is a defect unless explicitly authorized.

FROZEN means architectural changes are forbidden unless the packet explicitly exists to change architecture.

Constraint Strictness is independent of Risk Level.

A low-risk documentation packet may still be STRICT.

## Evidence Sensitivity

Evidence Sensitivity describes whether incorrect implementation can corrupt observatory evidence.

The standard must define at least:

- NONE
- LOW
- MEDIUM
- HIGH

Packets involving metrics, telemetry, validation, provenance, trust, admission, demotion, routing, or persistence must be evaluated for evidence sensitivity.

A packet that can cause false evidence, missing evidence, or misclassified evidence should be HIGH.

## Expected Blast Radius

Expected Blast Radius describes the expected scope of legitimate changes.

The standard must define at least:

- SINGLE_FILE
- MULTI_FILE_LOCAL
- CROSS_MODULE
- DATABASE
- WORKFLOW
- SYSTEMIC

Blast Radius is not a quality score.

A large blast radius may be correct if authorized.

An implementation exceeding expected blast radius must be treated as a routing and audit signal.

## Primary Skill Required

Primary Skill Required describes the main capability needed to complete the packet.

The standard must define a controlled vocabulary.

At minimum, include:

- SPECIFICATION_WRITING
- DATABASE_DESIGN
- MIGRATION_DESIGN
- TELEMETRY_PARSING
- VALIDATION_LOGIC
- CLI_IMPLEMENTATION
- TEST_DESIGN
- AUDIT_REASONING
- REFACTORING
- DOCUMENTATION_STRUCTURE
- ROUTING_POLICY
- UNKNOWN

This classification exists to support future model-task matching.

It must not be used as a model preference by itself.

## Audit Requirement

Audit Requirement describes how much independent review the packet requires.

The standard must define at least:

- NONE
- LIGHT
- STANDARD
- STRICT
- ADVERSARIAL

Packets that affect persistence, trust, admission, validation, telemetry, or routing must not use NONE.

ADVERSARIAL means the audit should actively search for subtle semantic failures, not only verify tests.

## Challenger Requirement

Challenger Requirement describes whether a second executor should be used.

The standard must define at least:

- NOT_REQUIRED
- OPTIONAL
- REQUIRED
- REQUIRED_DIVERSE

REQUIRED_DIVERSE means the challenger should be selected for different model behavior, not just similar capability.

High-risk or evidence-sensitive packets should prefer challenger execution until sufficient routing evidence exists.

## Routing Eligibility

Routing Eligibility describes whether historical evidence may be used to recommend an executor.

The standard must define at least:

- NOT_ELIGIBLE
- ELIGIBLE_WITH_HUMAN_REVIEW
- ELIGIBLE_FOR_RECOMMENDATION
- ELIGIBLE_FOR_AUTOMATED_SELECTION

FP-006 must state that early ForgePilot packets are not eligible for automated selection until sufficient admitted evidence exists.

Routing Eligibility must depend on admitted historical evidence, not anecdotal preference.

## Classification Record Requirements

The standard must define what a classification observation must include.

At minimum:

- packet_id
- task_class
- secondary_task_classes
- risk_level
- constraint_strictness
- evidence_sensitivity
- expected_blast_radius
- primary_skill_required
- audit_requirement
- challenger_requirement
- routing_eligibility
- classified_by
- classification_source
- rationale
- created_at

classification_source must distinguish human classification, model-assisted classification, and later automated classification.

## Correction and Revision

The standard must define how classification corrections work.

Classification corrections must be append-only observations.

Previous classifications must not be overwritten.

A correction must include:

- previous classification reference
- corrected fields
- reason
- actor
- created_at

Classification correction is itself evidence.

Frequent correction of classifications must be treated as a process signal.

## Routing Signal Relationship

FP-006 must define the relationship between task classification and future routing.

Task classification may inform routing only after being admitted as evidence.

A model win/loss must not be interpreted without task classification.

Example:

A model accepted on TESTING does not imply suitability for PERSISTENCE.

A model accepted on DOCUMENTATION does not imply suitability for TELEMETRY.

A model accepted after multiple scope corrections must not be treated the same as first-pass acceptance.

## Required Output

The executor must add a standards document defining packet task classification.

Suggested path:

- docs/task-classification-standards.md

The executor may update existing documentation only to add references to the new standards document.

## Acceptance Criteria

The result is acceptable if:

- A task classification standards document exists.
- It defines all required classification axes.
- It provides controlled vocabularies for each axis.
- It explains that axes are independent.
- It distinguishes task class from files touched.
- It defines correction as append-only observation.
- It connects classification to future routing signals.
- It states that classification does not itself authorize routing.
- It preserves the principle that only admitted evidence may influence observatory outputs.
- It does not add schema, CLI, routing, automation, dashboards, aggregation, or model selection behavior.

## Rejection Criteria

Reject the result if it:

- Adds SQLite schema changes.
- Adds CLI behavior.
- Adds routing logic.
- Adds model selection behavior.
- Adds autonomous execution behavior.
- Adds dashboards or reports.
- Treats classification as mutable state rather than observation.
- Collapses all classification axes into one status.
- Allows routing decisions without admitted evidence.
- Infers task class from executor output instead of packet intent.
- Optimizes for favorable model outcomes.
