# FP-008 — Classification and Outcome Persistence Implementation

## Task

Implement SQLite persistence for packet classification observations and model outcome observations.

## Goal

Move ForgePilot from standards-only classification/outcome artifacts toward durable, queryable evidence records.

FP-008 answers one question:

Can ForgePilot persist task classification and model outcome observations without using them for routing yet?

This packet implements persistence only.

It does not add model routing, model selection, scoring algorithms, automated recommendations, dashboards, reports, aggregation logic, autonomous execution, or observatory outputs.

## Governing Principles

This packet is constrained by:

* P01 — ForgePilot records observations, not narratives.
* P02 — Trust cannot be retroactively created.
* P03 — ForgePilot does not optimize for favorable outcomes.
* P04 — Only admitted evidence may influence observatory outputs.
* P05 — Do not build infrastructure for evidence that does not yet exist.
* P06 — Classification follows observation.

## Required Persistence Model

FP-008 must add append-only SQLite persistence for:

1. Packet classification observations
2. Packet classification correction observations
3. Model outcome observations
4. Model outcome correction observations

These records must be observations, not mutable current-state truth.

## Required Tables

The implementation must add a migration creating at least the following tables:

1. `packet_classification_observations`
2. `packet_classification_corrections`
3. `model_outcome_observations`
4. `model_outcome_corrections`

The migration must be idempotent.

The migration must not break existing FP-004 or FP-005 tables.

* task class
* risk level
* constraint strictness
* evidence sensitivity
* expected blast radius
* primary skill required
* audit requirement
* challenger requirement
* routing eligibility
* execution result
* verification result
* audit result

* correction type
* scope discipline
* semantic correctness
* invariant compliance
* human intervention
* comparison outcome
* non-blocking ambiguity
* root cause level
* routing signal eligibility
* trust tier
* validation state
* admission state

Enforcement may be implemented with TypeScript validation logic, SQLite `CHECK` constraints, or both.

If both are practical, prefer both.

## Append-Only Requirements

The implementation must preserve append-only behavior.

It must not allow:

* updating a classification observation in place
* deleting a classification observation
* updating an outcome observation in place
* deleting an outcome observation
* overwriting previous correction records

If the existing database layer does not currently expose update/delete methods, tests must still verify append-only behavior at the persistence API level.

## Trust and Admission Defaults

New classification and outcome observations must default to:

* `trust_tier`: appropriate low/default tier unless explicitly supplied
* `validation_state`: `VALID` only if all required fields are structurally valid
* `admission_state`: `PENDING` unless explicitly supplied by an authorized process

FP-008 must not automatically admit classification or outcome records as evidence.

FP-008 must not use these records for routing.

## CLI Scope

FP-008 may add minimal CLI commands only if necessary for manual inspection or testability.

Allowed CLI commands:

* record a classification observation
* record a model outcome observation
* list classification observations
* list model outcome observations

If CLI commands are added, they must not perform routing or recommendations.

CLI behavior is optional.

Persistence APIs and tests are required.

## Acceptance Criteria

The result is acceptable if:

* SQLite migration exists and is idempotent.
* Required classification and outcome tables exist.
* Classification observations are append-only.
* Classification corrections are append-only.
* Model outcome observations are append-only.
* Model outcome corrections are append-only.
* Required fields are persisted.
* Controlled vocabularies are enforced.
* New records are not automatically admitted.
* Existing FP-004 and FP-005 behavior is preserved.
* Tests cover the new persistence behavior.
* `pnpm typecheck` passes.
* `pnpm test` passes.
* No routing, model selection, scoring, aggregation, dashboards, reports, or autonomous execution behavior is added.

## Rejection Criteria

Reject the result if it:

* Adds routing logic.
* Adds model selection behavior.
* Adds scoring algorithms.
* Adds dashboards, reports, or aggregation.
* Automatically admits classification or outcome records as evidence.
* Allows classification or outcome records to influence routing.
* Mutates prior observations instead of appending corrections.
* Collapses task classification and model outcome into one table or one status field.
* Breaks FP-004 packet persistence.
* Breaks FP-005 telemetry ingestion.
* Treats telemetry as overriding audit outcome.
* Treats passing tests as audit acceptance.
* Treats comparison win as correctness.
* Optimizes for favorable model outcomes.
