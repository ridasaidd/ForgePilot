# FP-010 — SQLite Evidence Persistence

## Task

Implement a minimal SQLite persistence layer for ForgePilot evidence records.

## Goal

ForgePilot must be able to persist model run evidence in SQLite so that executor results, audit results, verification outcomes, provenance, artifact paths, and metrics references can be stored structurally instead of existing only as markdown or JSON files.

FP-010 answers one question:

Can ForgePilot store and retrieve evidence records for packet runs?

This packet creates persistence only.

It does not implement model routing, model selection, cost optimization, leaderboards, automatic task classification, evidence admission decisions, or observatory reporting.

## Scope

FP-010 may add:

- SQLite schema or migration for evidence records.
- Evidence persistence code.
- Evidence retrieval code.
- Tests for migration idempotence.
- Tests for inserting and reading evidence records.
- Minimal CLI or internal command support if the existing project structure requires it.

FP-010 must not add:

- Routing logic.
- Model ranking logic.
- Cost-efficiency scoring.
- Task classification.
- Automatic model recommendation.
- Leaderboards.
- Aggregated observatory reports.
- Changes to packet execution policy.
- Changes to model evaluation standards except documentation references required by persistence.

## Evidence Record Requirements

An evidence record must be able to represent, at minimum:

- `packet_id`
- `run_id`
- `model_id`
- `model_role`
- `branch`
- `commit_sha`
- `executor_result`
- `verification_result`
- `audit_result`
- `comparison_result`
- `metrics_path`
- `artifact_paths`
- `trust_tier`
- `validation_state`
- `admission_state`
- `created_at`

The exact schema may use normalized tables if appropriate, but the above information must be persistable and retrievable.

## State Constraints

FP-010 stores state values but does not decide them.

Validation and admission rules are reserved for FP-011.

Task classification and model comparison protocol are reserved for FP-012.

Therefore, FP-010 may persist values such as:

- `VALID`
- `INVALID`
- `INCOMPLETE`
- `DEFERRED`
- `NOT_EVALUATED`
- `REJECTED`
- `PENDING`
- `ADMITTED`
- `QUARANTINED`

But FP-010 must not implement the full policy that determines those values.

## Provenance Requirements

Persisted evidence must preserve enough provenance to trace a record back to:

- the packet,
- the run,
- the model,
- the role the model performed,
- the git branch,
- the git commit,
- the run artifacts,
- the metrics artifact.

A persisted evidence record must not replace the existing markdown artifacts in this packet.

Markdown and JSON run artifacts remain the human-readable evidence trail.

SQLite persistence becomes the structured evidence layer.

## Acceptance Criteria

1. A SQLite migration or schema update exists for evidence persistence.
2. The migration is idempotent.
3. Evidence records can be inserted.
4. Evidence records can be retrieved by packet ID.
5. Evidence records can be retrieved by run ID or equivalent unique run identifier.
6. Artifact paths can be persisted and retrieved.
7. Existing tests continue to pass.
8. New tests cover evidence insertion and retrieval.
9. The implementation does not add routing, ranking, task classification, leaderboards, or cost optimization.
10. The implementation does not remove or replace existing markdown or JSON artifacts.
