# FP-009 — Evidence Admission Persistence

## Status

Proposed

## Rationale

Rationale: `docs/rationale/FP-009-evidence-admission-rationale.md`

## Task

Implement SQLite persistence for evidence admission, admission review triggers, and admission invalidation events.

FP-009 defines how stored observations become admitted evidence without mutating the original observation rows.

## Goal

Make evidence admission structurally enforceable.

FP-009 answers one question:

> Which stored observations are currently eligible to be used as evidence?

This packet introduces append-only admission, review, and invalidation records.

It does not make evidence eligibility a mutable field on the original observation.

## Governing Scope Laws

FP-009 is governed by four scope laws:

1. Persistence is not admission.
2. Admission is an observation too.
3. The gate must be auditable by the same standards as what passes through it.
4. Invalidation begins with a recorded review trigger.

## Primary Invariant

Current evidence eligibility is derived from the valid event chain, never stored as mutable truth on the observation.

Observation rows must not contain fields such as:

```text
is_evidence = true
evidence_status = "ADMITTED"
```

Those fields collapse observation storage and evidence admission into one layer and are forbidden by this packet.

## Scope

FP-009 must add persistence support for:

```text
evidence_admission_events
admission_review_requests
admission_invalidation_events
```

These records must be append-only.

FP-009 may add query helpers or repository functions needed to write and read these records.

FP-009 may add tests verifying that evidence eligibility is derived from the event chain.

## Out of Scope

FP-009 must not implement:

* routing
* reporting
* dashboards
* aggregation
* model selection
* scoring logic
* policy optimization
* automated review scheduling
* mutation of classification observation rows
* mutation of outcome observation rows
* mutable evidence status fields on observation rows

## Event Model

### Stored Observations

FP-008 stores classification and outcome observations.

FP-009 does not redefine those observations.

FP-009 only references them.

A target observation must be identifiable by:

```text
target_observation_type
target_observation_id
```

Allowed initial observation types:

```text
classification_observation
outcome_observation
```

## Evidence Admission Events

An evidence admission event records that a stored observation was admitted for evidentiary use.

Admission must not mutate the target observation.

Required properties:

```text
id
target_observation_type
target_observation_id
admission_decision
admission_actor_type
admission_actor_id
admission_trust_tier
validation_state
provenance_complete
admission_basis
created_at
```

Allowed `admission_decision` values:

```text
ADMITTED
REJECTED
PENDING
```

Allowed `admission_actor_type` values:

```text
human_auditor
automated_validator
model_reviewer
cross_model_consensus
system
```

Allowed `admission_trust_tier` values:

```text
TIER_0
TIER_1
TIER_2
TIER_3
```

Allowed `validation_state` values:

```text
VALID
INVALID
INCOMPLETE
DEFERRED
```

Admission events are valid evidence gates only when:

```text
admission_decision = ADMITTED
validation_state = VALID
provenance_complete = true
admission_trust_tier is sufficient for the intended evidence use
```

FP-009 does not define routing-grade thresholds. It only preserves the trust tier needed for later policy decisions.

## Admission Review Requests

An admission review request records that an admission event has been challenged, flagged, or selected for review.

A review request does not invalidate anything by itself.

Required properties:

```text
id
target_admission_event_id
review_trigger_type
requested_by_actor_type
requested_by_actor_id
review_reason
validation_state
provenance_complete
created_at
```

Allowed `review_trigger_type` values:

```text
MANUAL_AUDITOR_FLAG
PROVENANCE_GAP_DISCOVERED
ACTOR_MISMATCH
VALIDATION_RULE_CHANGED
SOURCE_OBSERVATION_QUARANTINED
CONFLICTING_ADMISSION_EVENT
ROUTING_OUTPUT_ANOMALY
SYSTEM_INTEGRITY_CHECK
```

For V1, only `MANUAL_AUDITOR_FLAG` must be supported by normal write paths.

The other values may be defined for forward compatibility.

## Admission Invalidation Events

An admission invalidation event records that a previous admission event no longer grants evidentiary authority.

Invalidation targets the admission event, not the underlying observation.

Required properties:

```text
id
target_admission_event_id
review_request_id
invalidation_decision
invalidated_by_actor_type
invalidated_by_actor_id
invalidation_reason
validation_state
provenance_complete
created_at
```

Allowed `invalidation_decision` values:

```text
INVALIDATED
QUARANTINED
NO_ACTION
DEFERRED
```

Only these decisions remove or suspend evidentiary authority:

```text
INVALIDATED
QUARANTINED
```

`NO_ACTION` records that review occurred but the admission remains effective.

`DEFERRED` records that review could not yet reach a decision.

An invalidation event must reference an existing admission review request.

No invalidation may occur without a recorded review trigger.

## Derived Evidence Eligibility

Evidence eligibility must be derived from the event chain.

A stored observation is currently eligible as evidence only when:

1. A valid admission event exists for the observation.
2. The admission event has `admission_decision = ADMITTED`.
3. The admission event has `validation_state = VALID`.
4. The admission event has `provenance_complete = true`.
5. No later valid invalidation event defeats that admission.
6. The admission trust tier is sufficient for the intended consumer.

A later invalidation event defeats an admission when:

```text
target_admission_event_id matches the admission event
validation_state = VALID
provenance_complete = true
invalidation_decision is INVALIDATED or QUARANTINED
```

## Required Tests

FP-009 implementation must include tests proving:

1. Evidence admission events can be persisted.
2. Admission review requests can be persisted.
3. Admission invalidation events can be persisted.
4. Admission invalidation requires a review request.
5. Evidence eligibility is derived from the event chain.
6. Observation rows are not mutated when admitted.
7. Observation rows are not mutated when admission is invalidated.
8. Mutable evidence-state fields are not added to classification or outcome observation tables.
9. A valid admission with no defeating invalidation makes an observation eligible.
10. A later valid invalidation makes the same observation ineligible.
11. `NO_ACTION` review outcomes do not defeat admission.
12. Invalid or provenance-incomplete admission events do not grant eligibility.

## Acceptance Criteria

The packet is accepted only if:

1. The implementation adds append-only persistence for admission events.
2. The implementation adds append-only persistence for review requests.
3. The implementation adds append-only persistence for invalidation events.
4. Evidence eligibility is computed from events rather than stored on observations.
5. FP-008 observation tables remain immutable with respect to evidentiary status.
6. No `is_evidence`, `evidence_status`, or equivalent mutable state field is added to observation rows.
7. Tests cover admission, review, invalidation, and derived eligibility.
8. Existing tests continue to pass.
9. The implementation does not introduce routing, dashboards, aggregation, or model-selection behavior.
