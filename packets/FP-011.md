# FP-011 — Metrics Validation and Admission Integration

## Task

Implement validation and admission integration for persisted ForgePilot evidence records.

## Goal

ForgePilot must be able to evaluate persisted evidence records against existing trust, validation, provenance, and admission standards so that records can be classified as structurally valid, incomplete, deferred, invalid, pending, admitted, rejected, or quarantined without collapsing those concepts into one status field.

FP-011 answers one question:

Can a persisted evidence record be evaluated for evidence use without mutating the original observation?

This packet integrates validation and admission behavior with the SQLite evidence persistence added by FP-010.

It does not implement model routing, model selection, cost optimization, leaderboards, automatic task classification, model comparison matrices, dashboards, or observatory reporting.

## Scope

FP-011 may add:

- Validation functions for persisted evidence records.
- Admission integration functions for persisted evidence records.
- SQLite schema or migration updates if needed to record validation/admission evaluation events.
- Tests for validation outcomes.
- Tests for admission eligibility outcomes.
- Tests that original evidence records are not mutated by validation or admission events.
- Tests that trust tier, validation state, and admission state remain independent axes.
- Minimal CLI or internal command support if required by existing project structure.

FP-011 must not add:

- Model routing logic.
- Model ranking logic.
- Cost-efficiency scoring.
- Task classification.
- Automatic model recommendation.
- Leaderboards.
- Aggregated observatory reports.
- Dashboard output.
- Local model benchmarking.
- Changes to packet execution policy.

## Relationship to FP-010

FP-010 created structured evidence persistence.

FP-011 must build on that persistence layer.

FP-010 evidence records store values such as:

- `trust_tier`
- `validation_state`
- `admission_state`
- `artifact_paths`
- `metrics_path`
- provenance fields

FP-011 evaluates whether those records are safe to use as evidence.

FP-011 must not replace the FP-010 evidence records table as the source of persisted evidence records.

## Core Rule

Validation and admission must be recorded as observations or derived evaluations.

They must not retroactively mutate the original evidence record’s intent or provenance.

The original persisted evidence record remains historical evidence.

If a record is later rejected, admitted, or quarantined, that status must be represented through an append-only evaluation/event or a derived read model, not by rewriting the original record as if it had always had that state.

## State Axis Requirements

FP-011 must preserve the distinction between:

1. Trust Tier
2. Validation State
3. Admission State

These axes must not be collapsed into one status field.

### Trust Tier

Valid values:

- `TIER_0_UNTRUSTED`
- `TIER_1_SELF_REPORTED`
- `TIER_2_VERIFIED_ARTIFACT`
- `TIER_3_REPRODUCIBLE`

Trust tier describes source/provenance confidence.

### Validation State

Valid values:

- `VALID`
- `INVALID`
- `INCOMPLETE`
- `DEFERRED`

Validation state describes whether the record is structurally and logically usable.

`NOT_EVALUATED` must not be a validation state.

### Admission State

Valid values:

- `NOT_EVALUATED`
- `REJECTED`
- `PENDING`
- `ADMITTED`
- `QUARANTINED`

Admission state describes whether a record may influence observatory outputs.

## Minimum Validation Rules

FP-011 must be able to classify an evidence record according to these minimum rules.

### VALID

A record may be classified as `VALID` only if:

- required identity fields are present,
- required provenance fields are present,
- artifact paths are valid JSON array data,
- state-axis values are within the allowed vocabularies,
- the record is not logically contradictory.

### INCOMPLETE

A record should be classified as `INCOMPLETE` if required evidence is missing but the record is not logically invalid.

Examples:

- missing model identity,
- missing role,
- missing commit hash,
- missing metrics path,
- missing artifact paths.

### INVALID

A record should be classified as `INVALID` if it is structurally or logically contradictory.

Examples:

- malformed artifact path JSON,
- invalid state vocabulary,
- impossible state combination explicitly prohibited by standards,
- missing packet/run identity.

### DEFERRED

A record should be classified as `DEFERRED` if it cannot yet be evaluated because prerequisite data does not exist.

Examples:

- record references an expected artifact that is not available yet,
- admission depends on an audit or verification artifact that has not been recorded yet.

## Minimum Admission Rules

FP-011 must be able to derive whether an evidence record is eligible for admission.

At minimum:

A record may be `ADMITTED` only if:

- validation state is `VALID`,
- provenance is sufficient,
- trust tier is not `TIER_0_UNTRUSTED`,
- required verification or audit evidence is present where applicable.

A record must be `REJECTED` if:

- validation state is `INVALID`,
- provenance is contradictory,
- required evidence is known to be false or malformed.

A record may remain `PENDING` if:

- it is structurally valid but still requires review,
- trust tier is insufficient for automatic admission,
- human or cross-model audit is required.

A record may become `QUARANTINED` if:

- it was previously admitted but later invalidated,
- provenance becomes compromised,
- artifact integrity is later contradicted.

A record remains `NOT_EVALUATED` if no admission evaluation has been performed.

## Append-Only Requirement

FP-011 must not mutate original evidence records to simulate admission history.

Allowed approaches include:

- an append-only `evidence_record_validation_events` table,
- an append-only `evidence_record_admission_events` table,
- derived query functions that compute current validation/admission state from event history,
- another append-only design consistent with existing ForgePilot persistence standards.

If FP-011 stores current derived state, it must be clearly derived and must not replace the event history as the source of truth.

## Provenance Requirements

Validation/admission evaluation must preserve enough provenance to trace:

- which evidence record was evaluated,
- what validation/admission decision was made,
- which actor or process made the decision,
- when the decision was made,
- why the decision was made,
- whether the decision was automatic, model-generated, or human-reviewed.

## Acceptance Criteria

1. Validation logic exists for FP-010 evidence records.
2. Admission integration exists for FP-010 evidence records.
3. Validation and admission are represented without mutating original evidence records as historical observations.
4. Trust tier, validation state, and admission state remain independent axes.
5. `NOT_EVALUATED` is not accepted as a validation state.
6. `NOT_EVALUATED` remains valid as an admission state.
7. Valid records can be identified.
8. Incomplete records can be identified.
9. Invalid records can be identified.
10. Deferred records can be identified.
11. Admission eligibility can be derived for valid evidence records.
12. Rejection can be derived for invalid evidence records.
13. Quarantine or invalidation behavior can be represented without deleting prior records.
14. Tests verify append-only behavior.
15. Tests verify that existing FP-004, FP-005, FP-008, FP-009, and FP-010 behavior is preserved.
16. Existing tests continue to pass.
17. The implementation does not add routing, ranking, task classification, leaderboards, dashboards, reports, cost optimization, or model recommendation.
