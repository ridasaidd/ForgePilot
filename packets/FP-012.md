# FP-012 — Task Classification and Model Comparison Protocol

## Task

Define and implement a protocol for classifying ForgePilot packets and comparing model executions in a structured, evidence-backed way.

## Goal

ForgePilot must be able to classify a packet's task characteristics and compare competing model executions without turning comparison into model ranking, routing, leaderboards, cost optimization, or automatic model selection.

FP-012 answers one question:

> What kind of task was this, and how should two accepted or rejected executions be compared as evidence?

### This packet builds on:

* **FP-008:** Classification and outcome persistence
* **FP-009:** Evidence admission persistence
* **FP-010:** SQLite evidence persistence
* **FP-011:** Validation and admission integration

FP-012 must make comparison records more structured, repeatable, and queryable while preserving ForgePilot's evidence-first architecture.

## Core Rule

> **Task classification is observation. Model comparison is observation.**
> Neither classification nor comparison may mutate packet intent, evidence records, validation state, admission state, or execution history. FP-012 records what was observed about a task and what was observed when comparing executions. It does not decide future routing.

---

## Task Classification Protocol

FP-012 must support classification of a packet across these axes:

### Axes & Valid Vocabularies

| Axis | Valid Values | Description |
| --- | --- | --- |
| **Task Class** | `DOCUMENTATION`, `PERSISTENCE`, `CLI`, `TESTING`, `REFACTOR`, `BUGFIX`, `ARCHITECTURE`, `EVALUATION`, `TELEMETRY`, `UNKNOWN` | Describes the dominant kind of work. |
| **Risk Level** | `LOW`, `MEDIUM`, `HIGH`, `CRITICAL` | Describes the potential consequence of an incorrect implementation. |
| **Blast Radius** | `LOCAL`, `MODULE`, `DATABASE`, `WORKFLOW`, `REPOSITORY`, `UNKNOWN` | Describes the expected area affected by the packet. |
| **Evidence Sensitivity** | `LOW`, `MEDIUM`, `HIGH` | Describes how important evidence quality is for evaluating the task. |
| **Audit Requirement** | `NONE`, `STANDARD`, `STRICT` | Describes how much independent review the packet needs. |
| **Comparison Requirement** | `NONE`, `OPTIONAL`, `REQUIRED` | Describes whether multiple model executions should be compared. |
| **Classification Source** | `HUMAN`, `MODEL`, `SYSTEM`, `DERIVED` | Records how the classification was produced. |

### Minimum Classification Fields

A task classification record must preserve:

* packet identity
* task class
* risk level
* blast radius
* evidence sensitivity
* audit requirement
* comparison requirement
* classification source
* actor or model identity
* rationale
* created timestamp

Classification records must be **append-only**. If a classification is corrected, a new correction or superseding observation must be recorded rather than mutating the old observation.

## Model Comparison Protocol

FP-012 must support structured comparison of two or more model executions for the same packet.

### Minimum Comparison Fields

A comparison record must preserve:

* packet identity
* compared execution or evidence identifiers
* compared model identifiers
* compared model roles
* comparison outcome
* comparison basis
* selected execution or model, if any
* reason for selection
* known defects per compared execution
* verification results per compared execution
* audit results per compared execution
* whether each compared evidence item was admitted, pending, rejected, quarantined, or not evaluated
* actor or model performing the comparison
* created timestamp

### Comparison Outcome

Valid values:

* `MODEL_A_SELECTED`
* `MODEL_B_SELECTED`
* `TIE`
* `BOTH_ACCEPTED`
* `BOTH_REJECTED`
* `INCONCLUSIVE`
* `DEFERRED`

> [!NOTE]
> A comparison outcome must not be treated as a global model ranking. It only applies to the packet and compared executions.

### Comparison Basis

Valid values:

* `CORRECTNESS`
* `SCOPE_DISCIPLINE`
* `TEST_PASS_RATE`
* `SCHEMA_INTEGRATION`
* `PACKET_ALIGNMENT`
* `EVIDENCE_QUALITY`
* `MAINTAINABILITY`
* `COMPOSITE`

The comparison basis must describe why the comparison outcome was selected.

### Evidence Rules

A comparison record may reference evidence in these states:

* `ADMITTED`
* `PENDING`
* `REJECTED`
* `QUARANTINED`
* `NOT_EVALUATED`

**Rules:**

1. The comparison must explicitly record the evidence admission state at comparison time.
2. Only admitted comparison records may influence observatory outputs.
3. A comparison record must not silently promote pending, rejected, quarantined, or not-evaluated evidence into admitted evidence.

---

## Append-Only Requirement

FP-012 must preserve append-only behavior.

### Allowed approaches include:

* append-only task classification event tables
* append-only model comparison event tables
* correction tables that reference prior observations
* derived read functions that compute current classification or comparison summary

### Disallowed approaches include:

* overwriting the original packet record
* mutating original evidence records
* mutating original execution records
* mutating original classification records to simulate history
* replacing FP-008, FP-009, FP-010, or FP-011 tables

---

## Acceptance Criteria

1. Task classification records can be persisted.
2. Model comparison records can be persisted.
3. Classification records are append-only.
4. Comparison records are append-only.
5. Classification supports all required axes and vocabularies.
6. Comparison supports all required outcome and basis vocabularies.
7. Comparison records can reference model/evidence/execution identifiers.
8. Comparison records preserve admission state at comparison time.
9. Comparison does not mutate evidence records.
10. Comparison does not mutate packet intent.
11. Comparison does not mutate validation or admission events.
12. Invalid classification vocabulary values are rejected.
13. Invalid comparison vocabulary values are rejected.
14. Derived query functions can retrieve classification history by packet.
15. Derived query functions can retrieve comparison history by packet.
16. The latest classification or comparison can be derived without deleting history.
17. Existing FP-004, FP-005, FP-008, FP-009, FP-010, and FP-011 behaviors are preserved.
18. Existing tests continue to pass.
19. No routing, ranking, recommendation, dashboard, report, cost optimization, or local benchmarking behavior is added.
