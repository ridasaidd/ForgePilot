# FP-007 — Model Outcome Recording Standards

## Task

Define standards for recording model execution outcomes, correction events, audit outcomes, comparison outcomes, and routing-relevant performance signals.

## Goal

Ensure ForgePilot records model performance as structured evidence instead of narrative impressions.

FP-007 answers one question:

How did the model perform on this classified packet?

This packet defines standards only.

It does not add SQLite storage, CLI behavior, routing logic, model selection, scoring algorithms, aggregation, dashboards, reporting, autonomous execution, or schema implementation.

## Governing Principles

This packet is constrained by:

* P01 — ForgePilot records observations, not narratives.
* P02 — Trust cannot be retroactively created.
* P03 — ForgePilot does not optimize for favorable outcomes.
* P04 — Only admitted evidence may influence observatory outputs.
* P05 — Do not build infrastructure for evidence that does not yet exist.
* P06 — Classification follows observation.

## Core Model

Model outcome recording must distinguish:

1. Execution completion
2. Verification result
3. Audit result
4. Correction history
5. Comparison outcome
6. Routing signal value

These must not be collapsed into one status field.

A model may complete execution but fail audit.

A model may pass tests but violate packet scope.

A model may be accepted after correction but not count as first-pass success.

A model may lose comparison while still producing an accepted result.

A model may provide useful signal without producing mergeable evidence.

## Required Outcome Axes

FP-007 must define standards for at least the following outcome axes:

1. Execution Result
2. Verification Result
3. Audit Result
4. First-Pass Success
5. Correction Count
6. Correction Type
7. Scope Discipline
8. Semantic Correctness
9. Invariant Compliance
10. Human Intervention
11. Comparison Outcome
12. Non-Blocking Ambiguity
13. Root Cause Level
14. Routing Signal Eligibility

## Execution Result

Execution Result describes whether the model completed the requested execution attempt.

The standard must define at least:

* NOT_STARTED
* RUNNING
* COMPLETED
* FAILED
* ABORTED

Execution Result must not imply correctness.

A completed execution may still be rejected.

A failed execution may still provide useful diagnostic signal.

## Verification Result

Verification Result describes mechanical checks run after execution.

The standard must define at least:

* NOT_RUN
* PASSED
* FAILED
* PARTIAL
* BLOCKED

Verification Result may include:

* typecheck result
* test result
* lint result
* migration idempotence result
* required file existence checks
* forbidden file modification checks
* artifact existence checks

Verification Result must record command output or references to command output.

Verification Result must not be treated as audit acceptance.

Passing tests is not equivalent to acceptance.

## Audit Result

Audit Result describes independent review of the execution output against the packet.

The standard must define at least:

* NOT_AUDITED
* ACCEPTED
* REJECTED
* ACCEPTED_WITH_NOTES
* BLOCKED

Audit Result must be independent from Execution Result and Verification Result.

Audit Result must include:

* auditor identity
* auditor model or human actor
* audited commit or artifact reference
* blocking issues
* non-blocking notes
* root cause assessment
* required fix packet, if any

## First-Pass Success

First-Pass Success describes whether the execution was accepted without correction.

The standard must define:

* TRUE
* FALSE
* UNKNOWN
* NOT_APPLICABLE

First-Pass Success is TRUE only when:

* the execution completed
* verification passed or was not required
* audit accepted the result
* no corrective execution was required
* no packet-scope fix was required before acceptance

A model accepted after correction must not be treated as first-pass success.

First-Pass Success must be recorded separately from final acceptance.

## Correction Count

Correction Count records the number of required corrective cycles after the initial execution.

The standard must define correction count as a non-negative integer or UNKNOWN.

Correction Count must include only corrections required to satisfy the packet.

It must not include unrelated manual edits, formatting-only edits, or post-merge cleanup unless those edits were required for acceptance.

Correction Count must distinguish:

* executor self-correction
* auditor-requested correction
* human-requested correction
* process correction
* packet correction

## Correction Type

Correction Type describes what kind of correction was required.

The standard must define a controlled vocabulary.

At minimum, include:

* NONE
* SCOPE_CORRECTION
* SEMANTIC_CORRECTION
* MECHANICAL_CORRECTION
* TEST_CORRECTION
* ARTIFACT_CORRECTION
* PACKET_CORRECTION
* PROCESS_CORRECTION
* HUMAN_OVERRIDE
* UNKNOWN

A single execution may have multiple correction types.

Correction Type must be append-only.

Correction Type must not be overwritten after later acceptance.

## Scope Discipline

Scope Discipline describes whether the model stayed inside the packet’s authorized boundaries.

The standard must define at least:

* WITHIN_SCOPE
* MINOR_SCOPE_DRIFT
* MAJOR_SCOPE_DRIFT
* SCOPE_VIOLATION
* UNKNOWN

Scope Discipline must consider:

* unauthorized files changed
* unauthorized behavior added
* architectural changes added without permission
* extra CLI/schema/routing behavior added
* task expansion beyond packet intent

Scope Discipline must not be inferred only from file count.

A large change can be within scope if authorized.

A small change can be a scope violation if unauthorized.

## Semantic Correctness

Semantic Correctness describes whether the implementation satisfied the packet’s meaning, not merely its surface requirements.

The standard must define at least:

* CORRECT
* PARTIALLY_CORRECT
* INCORRECT
* NOT_EVALUATED
* UNKNOWN

Semantic Correctness must consider:

* packet intent
* acceptance criteria
* rejection criteria
* governing principles
* edge cases
* consistency with existing standards
* whether tests actually prove the required behavior

Semantic Correctness must not be reduced to passing tests.

## Invariant Compliance

Invariant Compliance describes whether the execution preserved required system invariants.

The standard must define at least:

* COMPLIANT
* VIOLATED
* NOT_CHECKED
* NOT_APPLICABLE
* UNKNOWN

Invariants may include:

* append-only event history
* immutable packet intent
* no automatic evidence admission
* explicit mapping requirements
* missing evidence remains null
* trust/admission/validation axes remain independent
* environment owns truth
* classifications are observations, not mutable state

Invariant violations must be treated as high-value routing and audit signals.

## Human Intervention

Human Intervention records whether a human had to correct, redirect, reinterpret, or rescue the execution.

The standard must define at least:

* NONE
* REVIEW_ONLY
* CLARIFICATION
* CORRECTION_REQUEST
* MANUAL_FIX
* OVERRIDE
* UNKNOWN

Human Intervention must distinguish normal review from corrective intervention.

Human review after execution does not automatically count as human intervention unless it changed the execution path or acceptance outcome.

## Comparison Outcome

Comparison Outcome records how an execution performed relative to another execution of the same packet.

The standard must define at least:

* NOT_COMPARED
* WIN
* LOSS
* TIE
* ACCEPTED_NOT_SELECTED
* REJECTED
* INCONCLUSIVE

Comparison Outcome must not replace Audit Result.

Two executions may both be accepted, while one is selected as the merge candidate.

A losing execution may still provide admitted evidence.

A comparison outcome must include the compared execution references.

## Non-Blocking Ambiguity

Non-Blocking Ambiguity records ambiguity discovered during audit or comparison that did not require rejection.

The standard must define at least:

* NONE
* PRESENT
* UNKNOWN

If PRESENT, the record must include:

* ambiguity description
* affected files or sections
* reason it was non-blocking
* whether it should influence future routing

Non-blocking ambiguity is routing signal, not failure by itself.

## Root Cause Level

Root Cause Level records the apparent source of a failure, correction, or ambiguity.

The standard must define at least:

* NONE
* PACKET
* EXECUTOR
* AUDITOR
* PROCESS
* ENVIRONMENT
* TOOLING
* UNKNOWN

Root Cause Level must not assign blame without evidence.

Root Cause Level must be allowed to remain UNKNOWN.

Root cause classification must be corrected by append-only observation if later evidence changes the assessment.

## Routing Signal Eligibility

Routing Signal Eligibility describes whether the outcome record may influence future routing analysis.

The standard must define at least:

* NOT_ELIGIBLE
* ELIGIBLE_AS_SIGNAL
* ELIGIBLE_AS_EVIDENCE
* QUARANTINED

Routing Signal Eligibility must depend on:

* task classification availability
* provenance completeness
* audit result
* validation state
* admission state
* telemetry completeness
* correction history completeness

Outcome records must not influence routing unless admitted as evidence.

Outcome records may remain useful as signal even when not eligible as evidence.

## Outcome Record Requirements

The standard must define what a model outcome record must include.

At minimum:

* packet_id
* packet_classification_reference
* execution_id
* executor_model
* executor_provider
* execution_result
* verification_result
* audit_result
* first_pass_success
* correction_count
* correction_types
* scope_discipline
* semantic_correctness
* invariant_compliance
* human_intervention
* comparison_outcome
* compared_execution_references
* non_blocking_ambiguity
* root_cause_level
* routing_signal_eligibility
* telemetry_reference
* audit_reference
* created_at

## Correction and Revision

Outcome corrections must be append-only observations.

Previous outcome records must not be overwritten.

A correction must include:

* previous outcome reference
* corrected fields
* reason
* actor
* created_at

Outcome correction is itself evidence.

Frequent correction of outcome records must be treated as a process signal.

## Relationship to Task Classification

FP-007 must define the relationship between task classification and outcome recording.

Outcome records must reference task classification when available.

A model outcome must not be interpreted without the packet’s task classification.

Example:

A model win on STANDARDS does not imply suitability for SCHEMA.

A model first-pass success on DOCUMENTATION does not imply suitability for TELEMETRY.

A model accepted after correction must not be treated the same as a first-pass success.

## Relationship to Telemetry

FP-007 must define the relationship between model outcome records and telemetry records.

Telemetry records may provide token, cost, latency, and session evidence.

Outcome records describe performance and judgment.

Telemetry must not override audit outcome.

Audit outcome must not fabricate telemetry.

If telemetry is missing, cost and token fields must remain null or unknown.

## Required Output

The executor must add a standards document defining model outcome recording.

Suggested path:

* docs/model-outcome-recording-standards.md

The executor may update existing documentation only to add references to the new standards document.

## Acceptance Criteria

The result is acceptable if:

* A model outcome recording standards document exists.
* It defines all required outcome axes.
* It provides controlled vocabularies for each axis.
* It distinguishes execution completion, verification, audit, correction, comparison, and routing signal eligibility.
* It defines first-pass success separately from final acceptance.
* It defines correction count and correction type.
* It defines scope discipline and semantic correctness.
* It defines invariant compliance.
* It defines human intervention.
* It defines comparison outcome.
* It defines non-blocking ambiguity.
* It defines root cause level.
* It defines routing signal eligibility.
* It defines minimum outcome record fields.
* It defines outcome correction as append-only observation.
* It connects outcome recording to task classification.
* It connects outcome recording to telemetry without allowing telemetry to override audit.
* It preserves the principle that only admitted evidence may influence observatory outputs.
* It does not add schema, CLI, routing, automation, dashboards, aggregation, reporting, scoring, or model selection behavior.

## Rejection Criteria

Reject the result if it:

* Adds SQLite schema changes.
* Adds CLI behavior.
* Adds routing logic.
* Adds model selection behavior.
* Adds scoring algorithms.
* Adds autonomous execution behavior.
* Adds dashboards, reports, or aggregation.
* Treats passing tests as equivalent to audit acceptance.
* Treats final acceptance as equivalent to first-pass success.
* Treats comparison win as equivalent to correctness.
* Treats telemetry as overriding audit result.
* Allows outcome records to influence routing without admission.
* Collapses outcome axes into one status.
* Mutates prior outcome records instead of appending corrections.
* Optimizes for favorable model outcomes.
