# Telemetry Authority and Field Ownership Standards

## Purpose

This document defines which system is authoritative for each ForgePilot metrics field.

These standards answer one question:

**Who is allowed to write each metrics field?**

This document defines authority and ownership policy only. It does not implement telemetry collection, OpenCode integration, metrics persistence, database schema, CLI commands, routing logic, aggregation logic, or workflow orchestration.

---

## Governing Principles

These standards are constrained by:

- **P01** — ForgePilot records observations, not narratives.
- **P02** — Trust cannot be retroactively created.
- **P03** — ForgePilot does not optimize for favorable outcomes.
- **P04** — Only admitted evidence may influence observatory outputs.
- **P06** — Classification follows observation.

---

## Core Concepts

### Field Authority

Field Authority is the exclusive right to assign a value to a metrics field. Only the authoritative source may write the field. Non-authoritative sources must not guess, infer, or fabricate values for fields they do not own.

### Field Owner

Field Owner is the lifecycle phase responsible for populating a field. The owner determines when the field is written and under what conditions. Each field has exactly one owner at any point in the lifecycle.

### Field Source

Field Source is the origin of the field's value. Sources include infrastructure telemetry exports, executor observations, auditor judgments, comparison outcomes, human annotations, and derived computations.

### Field Writer

Field Writer is the actor or process that physically writes the field value to the metrics artifact. The writer must be authorized by the field owner and must write only values from the field source.

### Infrastructure-Owned Telemetry

Infrastructure-owned telemetry consists of measurements produced by the execution environment, tooling, and platform. These values are observed by infrastructure, not generated by models. Examples include session identifiers, provider identifiers, timestamps, token counts, and cost estimates.

Infrastructure telemetry is the highest-trust source for operational measurements. Models must not guess infrastructure telemetry.

### Executor-Owned Observations

Executor-owned observations are values recorded by the executor model during task execution. These include execution notes, implementation decisions, and self-reported outcomes. Executor observations are self-reported and subject to trust tier classification.

### Auditor-Owned Judgments

Auditor-owned judgments are determinations made by the auditor model or human auditor during review. These include audit results, root cause classifications, and quality assessments. Auditor judgments are authoritative for audit-phase fields.

### Comparison-Owned Outcomes

Comparison-owned outcomes are determinations made during the model comparison phase. These include comparison results and winner selections. Only the comparison phase may write comparison-owned fields.

### Human-Entered Annotations

Human-entered annotations are values provided by human operators. These include manual corrections, contextual notes, and override decisions. Human-entered telemetry is lower trust than infrastructure-exported telemetry and must be clearly marked as human-sourced.

### Unknown/Unavailable Values

Unknown or unavailable values are fields whose values cannot be determined at the time of recording. These must remain `null` for numeric and boolean fields or `""` for string fields. Models must not guess unknown values. Missing values must be preserved as missing rather than fabricated.

## Authority Classes

### EXECUTOR

The executor model or executor process owns the field. Executor-owned fields are populated during executor run phases. Executor observations are self-reported and subject to TIER_1 trust classification unless verified by artifacts.

### AUDITOR

The auditor model or human auditor owns the field. Auditor-owned fields are populated during the audit phase. Auditor judgments are authoritative for audit determinations.

### COMPARISON_PHASE

The comparison process owns the field. Comparison-owned fields are populated only after all executor runs complete and comparison evaluation finishes. Executors and auditors must not write comparison-owned fields.

### INFRASTRUCTURE

The execution environment, tooling, or platform owns the field. Infrastructure-owned fields contain measurements observed by infrastructure, not generated by models. Models must not guess infrastructure telemetry.

### OPENCODE_TELEMETRY

OpenCode session exports own the field when available. OpenCode telemetry provides infrastructure-grade measurements for token usage, cost, session identifiers, and provider information. When OpenCode telemetry is available, it is authoritative for token and cost fields.

### HUMAN_OPERATOR

A human operator owns the field. Human-owned fields contain manual annotations, corrections, or override decisions. Human-entered values are lower trust than infrastructure-exported values and must be clearly marked.

### DERIVED

The field is computed from other fields. Derived fields must identify their source fields and computation method. Derived fields inherit trust from their sources and must not elevate trust beyond the lowest-trust source field.

### UNKNOWN

The field authority is not yet determined or the field is not yet owned by any phase. Unknown authority fields must remain null or empty until an authoritative source provides a value.

## OpenCode Telemetry Evidence

OpenCode session exports may provide the following telemetry:

- session id
- project id
- directory
- model id
- provider id
- model variant
- cost
- input tokens
- output tokens
- reasoning tokens
- cache tokens
- created timestamp
- updated timestamp
- message-level token usage
- changed file summaries
- tool activity

### OpenCode Telemetry Classification

OpenCode telemetry is classified as infrastructure-grade evidence when:

1. The telemetry is exported directly from OpenCode session records.
2. The export is unmodified and preserves original values.
3. The session identifier links the telemetry to a specific execution.

OpenCode telemetry is authoritative for:

- Token counts (prompt, completion, reasoning, total)
- Cost estimates
- Session identifiers
- Provider identifiers
- Model identifiers and variants
- Timestamps

OpenCode telemetry is not authoritative for:

- Audit results
- Comparison outcomes
- Root cause classifications
- Quality judgments
- Human intervention flags

### OpenCode Telemetry Trust

OpenCode-exported telemetry is higher trust than model-estimated telemetry.

When OpenCode telemetry is available, token and cost fields must use OpenCode values.

When OpenCode telemetry is unavailable, token and cost fields must remain `null`. Models must not estimate or guess token counts or costs.

## Authority Rules

### Models Must Not Guess Infrastructure Telemetry

Executor models and auditor models must not fabricate, estimate, or infer infrastructure-owned measurements. If infrastructure telemetry is unavailable, the field must remain `null` or `""` according to field type.

**Prevents:** Fabricated telemetry contaminating evidence. (P01, P04)

### Token and Cost Fields Require OpenCode Telemetry

Token counts (`prompt_tokens`, `completion_tokens`, `reasoning_tokens`, `total_tokens`) and cost estimates (`estimated_cost`) must come from OpenCode telemetry exports when available.

When OpenCode telemetry is unavailable, these fields must remain `null`.

**Prevents:** Model-estimated token counts from being treated as evidence. (P01, P02)

### Infrastructure Observations Are Authoritative

Session ID, provider, model variant, timestamps, and cost are infrastructure observations. Infrastructure is the authoritative source for these measurements.

**Prevents:** Models from claiming infrastructure measurements without infrastructure evidence. (P01, P04)

### Models May Reference Telemetry Only From Authoritative Sources

A model may reference telemetry values only if those values were provided by an authoritative source. Models must not reference telemetry they generated themselves unless clearly marked as self-reported.

**Prevents:** Circular telemetry claims. (P04)

### Missing Telemetry Must Remain Null or Empty

When telemetry is unavailable, the field must remain `null` for numeric and boolean fields or `""` for string fields. Missing telemetry must not be replaced with estimates, defaults, or placeholders that suggest a value exists.

**Prevents:** Fabricated values from being treated as observations. (P01, P03)

### Human-Entered Telemetry Is Lower Trust

Human-entered telemetry is classified as lower trust than infrastructure-exported telemetry. Human annotations must be clearly marked as human-sourced and must not be promoted to infrastructure trust.

**Prevents:** Human estimates from being treated as infrastructure measurements. (P02)

### Field Authority Must Be Recorded

Field authority must be documented before persistence rules are finalized. Each field must have a clearly identified authority class, owner phase, source, and writer.

**Prevents:** Ambiguous field ownership from causing conflicting writes. (P01)

### Fields May Be Updated Only By Owning Phase

A field may be written or updated only by its owning lifecycle phase. Later phases must not overwrite earlier-phase fields unless explicitly authorized by escalation or correction rules.

**Prevents:** Lifecycle phase contamination. (P01)

### Derived Fields Must Identify Sources

Derived fields must document their source fields and computation method. Derived fields inherit trust from their sources and must not claim higher trust than their lowest-trust source.

**Prevents:** Derived values from obscuring their provenance. (P02, P06)

### Conflicting Values Prefer Authoritative Source

When multiple values exist for the same field, the authoritative source takes precedence. Non-authoritative values must be discarded or marked as superseded.

**Prevents:** Conflicting evidence from contaminating observatory outputs. (P04)

## Unknown and Unavailable Value Rules

### When Values Are Unknown

A field value is unknown when:

- The measurement has not yet been taken.
- The measurement tool is unavailable.
- The lifecycle phase has not yet completed.
- The authoritative source has not provided a value.

### Recording Unknown Values

Unknown numeric fields must be recorded as `null`.

Unknown boolean fields must be recorded as `null`.

Unknown string fields must be recorded as `""`.

Unknown fields must not be recorded as `0`, `false`, `"unknown"`, `"N/A"`, or any other placeholder that suggests a value exists.

### Transitioning From Unknown

When an authoritative source later provides a value, the field may be updated from `null` or `""` to the provided value.

The update must be performed by the field owner during the appropriate lifecycle phase.

Updates must preserve the original unknown state in audit history if audit trails are maintained.

---

## Conflicting Value Rules

### When Conflicts Occur

Conflicts occur when:

- Multiple sources provide different values for the same field.
- A non-authoritative source writes a value before the authoritative source.
- Human annotation contradicts infrastructure telemetry.
- Derived computation contradicts direct measurement.

### Conflict Resolution

Conflicts are resolved by authority precedence:

1. **INFRASTRUCTURE** and **OPENCODE_TELEMETRY** take precedence over all other sources.
2. **AUDITOR** takes precedence over **EXECUTOR** for audit-phase fields.
3. **COMPARISON_PHASE** takes precedence for comparison-owned fields.
4. **HUMAN_OPERATOR** annotations are preserved but marked as lower trust.
5. **DERIVED** values defer to direct measurements when available.

### Conflict Recording

When a conflict is detected and resolved:

- The authoritative value is written to the field.
- The conflicting value is discarded or recorded in a separate conflict log if maintained.
- The resolution reason is documented.

Conflicts must not be silently resolved by choosing the most recent value or the most convenient value.

---

## Derived Field Rules

### Identifying Derived Fields

Derived fields are computed from one or more source fields. Examples include:

- `total_tokens` computed from `prompt_tokens` + `completion_tokens` + `reasoning_tokens`
- Aggregated metrics computed from individual run metrics
- Success rates computed from multiple run outcomes

### Derived Field Requirements

Derived fields must:

1. Identify all source fields used in the computation.
2. Document the computation method or formula.
3. Inherit trust from the lowest-trust source field.
4. Be recomputed when source fields change.
5. Be marked as derived in field metadata.

### Derived Field Trust

Derived fields must not claim higher trust than their sources. If any source field is TIER_1_SELF_REPORTED, the derived field must not be treated as TIER_2 or higher.

Derived fields are subject to the same validation and admission rules as direct measurements.

## Relationship to Other Standards

These standards complement:

- **Metrics Trust and Validation** (`docs/metrics-trust-and-validation.md`) — Defines trust tiers, validation states, and admission rules. Field authority determines which source may write a field; trust and validation determine whether the written value is admissible as evidence.

- **Model Evaluation Harness** (`docs/model-evaluation-harness.md`) — Defines evaluation methodology and metrics artifact schema. Field ownership determines which lifecycle phase populates each field in the metrics artifact.

- **Task Classification Standards** (`docs/task-classification-standards.md`) — Defines classification axes for packets including task class, risk level, and audit requirement. Classification records are themselves observations with defined field authority and ownership requirements.

Field authority must be established before trust tier assignment. A field written by a non-authoritative source fails provenance completeness requirements regardless of the value's correctness.
